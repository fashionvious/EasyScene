==============================================================================
LLM API 超时问题分析与解决思路
==============================================================================

【问题现象】
- 调用 qwen3.5-plus 模型时，请求在约60秒后超时（APITimeoutError: Request timed out.）
- qwen3.6-flash 模型可以正常调用，不会超时
- OpenAI SDK 内置了2次自动重试，每次重试也超时，总耗时约180秒后最终报错

==============================================================================
【原因分析】
==============================================================================

1. **qwen3.5-plus 模型推理速度慢于 flash 模型**
   - flash 系列模型是轻量/快速推理模型，响应时间通常在5-15秒
   - plus 系列模型是高能力模型，推理时间更长，通常需要30-120秒甚至更久
   - 对于长prompt（如首帧图提示词生成，prompt约500+字），plus模型可能需要60秒以上

2. **超时时间设置过短**
   - .env 中 LLM_TIMEOUT=60，但代码中默认值是120
   - 实际读取到的是60秒（.env优先）
   - 60秒对于 flash 够用，但对 plus 模型不够

3. **代码中硬编码了模型名，绕过了.env配置**
   - first_and_last_frame_agent.py 第204行：
     self.llm_client.think(messages=messages, model="qwen3.5-plus")
   - 这里强制使用 qwen3.5-plus，即使 .env 配置的是 flash 模型
   - 尾帧提示词生成（第240行）没有指定model，会用.env的flash模型

4. **使用非流式（stream=False）请求**
   - 非流式请求需要等服务端完全生成后才能返回
   - 流式请求可以边生成边接收，不容易触发读超时
   - 当前 think() 方法默认 stream=False

5. **OpenAI SDK 内置重试机制增加了总等待时间**
   - 日志显示 "Retrying request to /chat/completions in 0.457129 seconds"
   - SDK默认重试2次，每次都等60秒超时，总耗时 = 60*3 = 180秒
   - 用户看到的是最终失败，但已经等了3分钟

6. **网络因素**
   - dashscope API 服务器在阿里云，网络延迟一般不高
   - 但如果网络不稳定或高峰期，响应时间可能波动

==============================================================================
【解决思路】
==============================================================================

------ 方案1：增大超时时间（最简单）------
- 将 .env 中的 LLM_TIMEOUT 从 60 改为 180 或 300
- 优点：改动最小，一行配置
- 缺点：如果模型真的挂了，需要等更久才知道失败

------ 方案2：移除硬编码的模型名，统一使用 .env 配置------
- 删除 first_and_last_frame_agent.py 第204行的 model="qwen3.5-plus"
- 让所有LLM调用统一使用 .env 中配置的 LLM_MODEL_ID
- 优点：配置集中管理，灵活切换模型
- 缺点：如果确实需要不同步骤用不同模型，需要其他机制

------ 方案3：改用流式请求（stream=True）------
- 将 think() 调用改为 stream=True
- 流式请求一旦建立连接就开始接收数据，不容易触发读超时
- 优点：从根本上避免超时问题，用户体验更好
- 缺点：需要处理流式响应的拼接逻辑（代码中已有此逻辑）

------ 方案4：为不同模型设置不同的超时时间------
- 在 think() 方法中，根据模型名称动态调整超时
- 例如：flash 模型 60秒，plus 模型 180秒
- 优点：兼顾速度和可靠性
- 缺点：维护模型-超时映射表

------ 方案5：添加重试与降级逻辑------
- 在 think() 方法中捕获超时异常后：
  (a) 自动重试（增加超时时间）
  (b) 降级到 flash 模型重试
- 优点：提高鲁棒性，不会因单次超时而整体失败
- 缺点：代码复杂度增加

------ 方案6：减少 max_tokens 降低响应时间------
- 当前 max_tokens=2048，对于生成150字提示词来说偏大
- 可降低到 512 或 1024，模型生成更少token，响应更快
- 优点：简单有效
- 缺点：可能截断长回复（但提示词生成场景不太可能超512 token）

------ 方案7：配置 OpenAI SDK 的重试参数------
- 设置 max_retries=0 或减少重试次数
- 设置更合理的重试等待策略
- 优点：快速失败，不浪费3分钟等待
- 缺点：降低了网络抖动时的自动恢复能力

==============================================================================
【推荐组合方案】======
- 方案2（移除硬编码）+ 方案1（增大超时到180秒）+ 方案6（降低max_tokens到1024）
- 这是最小改动、最大收益的组合
- 如果仍偶发超时，再叠加方案3（改流式）或方案5（降级重试）
==============================================================================
